## Initial setup

Clone the repository and install dependencies:

```bash
git clone https://github.com/unkeyed/unkey
cd unkey
make install
```

## Configure depot

We use depot as build runner and right now that is not optional.

```bash
cp ./dev/.env.depot.example ./dev/.env.depot
# Edit ./dev/.env.depot with your Depot credentials
```

### Run dev mode

Start the full development setup:

```bash
make dev
```

You get:

- Tilt UI at `http://localhost:10350`
- Various services port-forwarded
- Dashboard at `http://localhost:3000`

### Dashboard-only development

If you are iterating on the dashboard and do not need the full Kubernetes stack, `make dashboard` is a lighter alternative. It runs the backend services in Docker via `docker compose` and then starts the Next.js dev server with hot reload. It does not require Tilt, minikube, or ctlptl.

Before the first run, make sure dependencies are installed:

```bash
make install
```

Then start the dashboard dev environment:

```bash
make dashboard
```

## Local HTTPS with Frontline (optional)

Set up local TLS for `*.unkey.local`:

1. Configure local DNS:

```bash
./dev/setup-wildcard-dns.sh
```

2. Start the minikube tunnel in another terminal:

```bash
make tunnel
```

3. Open the local domain:

```bash
open https://app.unkey.local
```

Tilt generates trusted TLS certificates using mkcert and Frontline terminates TLS on port 443.

## Stop the development environment

```bash
make down
```

## Environment configuration

### Local authentication

Set local auth mode in your `.env` file:

```plaintext
AUTH_PROVIDER="local"
```

### Optional services

WorkOS authentication:

```plaintext
AUTH_PROVIDER="workos"
WORKOS_CLIENT_ID=<your client ID>
WORKOS_API_KEY=<your API key>
WORKOS_COOKIE_PASSWORD=<your base64 password>
```

Stripe billing:

```plaintext
STRIPE_SECRET_KEY=<your Stripe secret key>
```

## Seed local data

```bash
make unkey dev seed local
```

This creates a fixed scaffold (workspace, root API, root key) suitable for booting the dashboard.

## Seed with Claude Code

For ad-hoc scenarios beyond `seed local` (e.g. "create a project with a deployment in every status", "make me an identity with rate limits"), the repo ships an MCP server at `bin/seed-mcp` and a `/seed` skill that teaches Claude Code how to use it without violating Unkey's must-create-together rules.

### Setup

Build the binary (this also produces the main `unkey` binary):

```bash
make build
```

Register the MCP with Claude Code:

```bash
claude mcp add unkey-seed -- $(pwd)/bin/seed-mcp
```

The `/seed` skill is checked in at `.claude/skills/seed/SKILL.md` and loads automatically when Claude Code starts in this repo.

### Use

Start Claude Code in the repo root and ask for what you need, e.g.:

- _"Create a workspace with a working API key for testing."_
- _"Set up a deployable app called payments in ws_local, ready to deploy to the local region."_
- _"Seed an identity 'cust_acme' with a rate limit of 100 req/min."_

The `/seed` skill auto-fires on seeding tasks; you can also invoke it explicitly with `/seed`. It encodes the rules the schema can't enforce — for example: an `apps` row needs both `production` and `preview` envs plus runtime/build/regional settings, and a `deployments` row in `ready` status needs paired `deployment_topology`, `instances`, `cilium_network_policies`, and `frontline_routes`. The live schema files at `pkg/mysql/schema/` are the source of truth for column definitions; the skill covers what they can't.

### Tools and safety

The MCP exposes `list_tables`, `sample_rows`, `query` (read-only), `insert`, `bulk_insert`, and `generate_key`. There is no `truncate` or any other destructive operation. The DSN is hardcoded to the local docker MySQL (`unkey:password@tcp(127.0.0.1:3306)/unkey`); the binary will not connect anywhere else. If you need to point it elsewhere, edit `x/seed-mcp/main.go` and rebuild.

## Test locally

Run Go tests with Bazel:

```bash
make test
```

Run a single Go test:

```bash
bazel test //pkg/cache:cache_test --test_filter=TestCacheName
```

Run TypeScript tests with pnpm:

```bash
pnpm --dir=web test
```
