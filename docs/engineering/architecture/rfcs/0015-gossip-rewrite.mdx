## Summary

Replace the current two-tier `memberlist` cluster with a single flat [Serf](https://github.com/hashicorp/serf) cluster spanning all regions over VPC peering. Rewrite the application-facing API around a generic event bus where any payload can be broadcast and subscribed to; cache invalidation becomes one event type among many. Delete the bridge election, the WAN NLB, and the LAN/WAN delegate split.

## Motivation

The current cluster ([`pkg/cluster`](https://github.com/unkeyed/unkey/tree/main/pkg/cluster)) is fragile in production. Concretely:

- **Two memberlists per node.** Every region runs a LAN pool, and one elected node also runs a WAN pool that gateways traffic between regions. This doubles the failure surface (two sockets, two queues, two delegates, two sets of seeds) and forces the relay logic in [`delegate_lan.go`](https://github.com/unkeyed/unkey/blob/main/pkg/cluster/delegate_lan.go) and [`delegate_wan.go`](https://github.com/unkeyed/unkey/blob/main/pkg/cluster/delegate_wan.go) to clone and re-marshal every cross-region message.
- **Bridge election is hand-rolled and lossy.** [`bridge.go`](https://github.com/unkeyed/unkey/blob/main/pkg/cluster/bridge.go) elects the lexicographically smallest node name as bridge. Promotions and demotions trigger `memberlist.Create`/`Shutdown` cycles that race with rolling deploys; the retry loop on socket reuse is a workaround, not a fix. During a transition, cross-region messages drop on the floor with no record.
- **WAN NLB plumbing is brittle.** The internet-facing NLB in [`service-gossip-wan.yaml`](https://github.com/unkeyed/unkey/blob/main/k8s/api/base/service-gossip-wan.yaml) health-checks port 7947 to find "the bridge pod". This conflates routing with election: re-elections must wait for NLB health checks to converge, and `WANAdvertiseAddr` has to be resolved manually in [`bridge.go`](https://github.com/unkeyed/unkey/blob/main/pkg/cluster/bridge.go) because memberlist will not accept a hostname.
- **One-shot, fire-and-forget messages.** `clusterBroadcast.Invalidates` always returns `false` and the queue retransmits a fixed number of times. There is no acknowledgement, no retention, no replay for nodes that join late or were partitioned. A missed cache invalidation is silently a stale read.
- **Transport is fused to one payload.** The `ClusterMessage` proto carries a `oneof payload` and the `MessageMux` dispatches on Go type assertions. Adding a new event type means editing the proto, the mux, and every delegate. There is no way to namespace by subsystem (api vs ctrl vs sentinel).
- **No cluster state.** Membership is the only replicated data. Anything that needs eventual consensus (leader for a cron, feature flags, tenant routing tables) cannot live here.

We have separately stood up VPC peering between the three deploy clusters and the global observability cluster ([`terraform/aws-vpc-peering/production/main.tf`](https://github.com/unkeyed/unkey/blob/main/terraform/aws-vpc-peering/production/main.tf)). A flat L3 mesh now exists. The two-tier design predates it and is no longer justified.

The expected outcome: one cluster object per process, one socket, one set of seeds, no bridge, no NLB. New event types ship by registering a handler. Late joiners and partitioned nodes converge on missed events automatically.

## Detailed design

### Transport: Serf over the peered mesh

Adopt [`hashicorp/serf`](https://github.com/hashicorp/serf) as the cluster layer. Serf is a thin layer on top of `memberlist` that adds the three primitives we are currently missing:

| Primitive | What it gives us |
|---|---|
| User events | Named, payload-carrying broadcasts with deduplication and a retention buffer for late joiners. |
| Queries | Request/response over gossip with an aggregated reply channel and a deadline. Useful for "who owns tenant X" or "drain status of all peers". |
| Coalesced membership events | One event per join/leave/update batch, not per-node. Removes the hot-path churn we currently get on rolling deploys. |

A single Serf cluster spans every region. Each pod joins the same logical cluster; seeds are the per-region headless service plus a small cross-region seed list (one stable hostname per region, resolved on join). VPC peering makes pod IPs routable, so memberlist's preferred IP-based addressing works without an NLB.

LAN vs WAN tuning is handled by Serf's profile selection (`WAN`, `LAN`, `Local`). Pick `WAN` for any cluster that includes cross-region peers; the higher timeouts cover trans-Atlantic RTT without false-positive failures. We accept ~5–15 s convergence on cross-region failures in exchange for deleting the bridge.

Encryption stays on (`SecretKey` from the existing config), now applied to all traffic uniformly instead of being configured twice.

### Event bus: generic, named topics

Replace the proto `oneof` payload with a transport-agnostic event envelope:

```
Event {
    topic: string         // "cache.invalidate", "feature_flag.update", ...
    id: string            // ULID, used for dedup
    source_region: string
    sender_node: string
    sent_at_ms: int64
    payload: bytes        // opaque to transport, decoded by the topic owner
}
```

The application API becomes:

```go
type Bus interface {
    Publish(ctx context.Context, topic string, payload proto.Message) error
    Subscribe(topic string, handler func(Event)) (unsubscribe func())
    Query(ctx context.Context, topic string, payload proto.Message) (<-chan Event, error)
    Members() []Member
    Close() error
}
```

`Query` fans out a Serf query to all members (or a tag-filtered subset) and returns a channel that receives replies as they arrive. The channel is closed when the context deadline fires or when Serf signals that all expected responses have arrived (`serf.QueryResponse.Finished()`), whichever comes first. Callers must not assume a particular number of responses: a partitioned member or one with no handler for the topic will not reply. The caller owns draining the channel before the context expires.

Cache invalidation is one subscriber:

```go
bus.Subscribe("cache.invalidate", func(e Event) {
    var inv cachev1.CacheInvalidationEvent
    if err := proto.Unmarshal(e.Payload, &inv); err != nil { ... }
    cache.Invalidate(inv)
})
```

The bus does not know about cache, sentinel, ctrl, or anything else. Each subsystem owns its own proto type and registers its own handler at startup. Adding a new event type requires no changes to `pkg/cluster`.

The envelope's `topic` is the dispatch key; `id` enables idempotent delivery; `payload` is opaque bytes so that two subsystems can evolve their schemas independently without recompiling the cluster package.

The serialized envelope plus payload must fit within `MaxUserEventSize` (default 512 bytes, configurable via `BusConfig`). Envelope overhead is approximately 80-120 bytes; payload protos must be designed to fit the remaining budget, or `MaxUserEventSize` must be raised explicitly in the bus config.

### Delivery: at-least-once

Best-effort fan-out is not enough for cache invalidation: a missed event means stale reads served until the entry's natural TTL expires. The bus guarantees **at-least-once delivery**, with handlers required to be idempotent (the envelope's `id` is the dedup key).

Mechanism, in order of escalation:

1. **Serf user events with retention.** Every publish goes out as a Serf user event with the 60-second retention buffer turned on. A peer that joins the cluster within that window receives buffered events replayed from Serf's `RecentIntents` queue. Individual packet loss within an established session is covered by Serf's gossip retransmission. A node that was fully partitioned does not receive missed user events on reconnect: Serf's anti-entropy reconciles membership state only, not user events. That gap is closed by the replay log below.
2. **Per-topic replay log on the publisher.** Each pod keeps the last 60 s of events it published in a local ring (RAM, bounded). On `NotifyJoin`, the new peer sends a Serf query (`bus.replay`) carrying the highest event id it has seen per source node. Publishers reply with anything missing. This closes the gap for partitions longer than the retention buffer when at least one publisher is reachable.
3. **Receiver-side dedup.** Each subscriber tracks recently seen `(source_node, id)` pairs (bounded LRU) and drops duplicates before invoking the handler.

This gives at-least-once for any event whose publisher (or any peer that received it) is still alive somewhere in the mesh within the recovery window. We do **not** offer durable, cross-restart at-least-once in v1 — if every replica of a publisher dies before its event propagates, the event is lost. For invalidation that is acceptable because the database write that triggered it is the source of truth and the next mutation will re-trigger eventually; for stronger guarantees we would need to write to an external durable log, which is out of scope here.

### Why not Raft / etcd

Raft (via [`hashicorp/raft`](https://github.com/hashicorp/raft)) and etcd both solve a different problem: linearizable replicated state with a quorum write path. We do not need linearizable cache invalidation — invalidation is already best-effort and the source of truth is the database. Forcing every invalidation through a Raft log adds:

- A quorum write on the hot path of every key mutation.
- A leader, which must be in one region; cross-region writes pay the RTT.
- Operational weight (snapshot management, log compaction, leader leases) for a workload where missing one event is recoverable.

Serf gives us eventually-consistent fan-out, which is the right consistency model for cache invalidation, feature flag pushes, and operational notifications. If we later need leader-elected coordination (singleton crons, schema migrations), we can run a small Raft group as a *separate* package on top of the same Serf membership — Raft for state, Serf for transport and discovery. We do not need it in v1.

### Topology: one flat cluster over VPC peering

```diagram
                ╭──────────────╮
                │  use1 pods   │──┐
                ╰──────┬───────╯  │
                       │           │  one Serf cluster, full mesh,
                       │           │  encrypted, profile=WAN
                ╭──────┴───────╮  │
                │  euc1 pods   │──┤
                ╰──────┬───────╯  │
                       │           │
                ╭──────┴───────╮  │
                │  usw2 pods   │──┘
                ╰──────────────╯
```

Seeds:

- **Intra-region**: the existing headless service [`unkey-api-gossip-lan`](https://github.com/unkeyed/unkey/blob/main/k8s/api/base/service-gossip-lan.yaml) keeps working; pods discover same-region peers via DNS A records.
- **Cross-region**: a small list of stable per-region hostnames resolved at join time. Three regions × two seed pods is enough; Serf only needs one reachable peer to converge.

The WAN NLB ([`service-gossip-wan.yaml`](https://github.com/unkeyed/unkey/blob/main/k8s/api/base/service-gossip-wan.yaml)) is deleted. So is port 7947 in the deployment. The pod exposes a single gossip port (7946 TCP+UDP) that is reachable from every peered VPC.

Security posture stays the same: all traffic is encrypted with `UNKEY_GOSSIP_SECRET_KEY`, and the security group rule for 7946 is restricted to the peered VPC CIDR ranges instead of being public.

### Membership of the cluster

The first cluster includes the API frontline pods. Once stable, the same binary-level `Bus` is embedded into the ctrl plane and any other long-lived service that benefits from event fan-out (sentinel, future schedulers). They all join the same Serf cluster. Tag each member with role metadata (`role=api`, `role=ctrl`) so subscribers can filter or address subsets via Serf queries.

This is the "one big fat cluster" the design calls for. It is safe because:

- Serf scales to thousands of members with `WAN` timings.
- Topics namespace events, and **subscription is per-process**. A pod only runs handlers for the topics it explicitly subscribes to; it pays the cost of receiving and decoding the envelope, but nothing more. Cross-role subscriptions are intentional and supported — an api pod can subscribe to `ctrl.deployment.changed` to pre-warm a route table, for example.
- All traffic is encrypted with the same shared key.

### Member metadata (tags)

Serf lets each member advertise a small key=value map that every peer can read. We standardise the keys up front so subscribers can filter members and target queries deterministically. The agreed schema for v1:

| Tag | Example | Set by |
|---|---|---|
| `role` | `api`, `ctrl`, `sentinel` | Process at startup |
| `region` | `us-east-1` | Env / overlay |
| `version` | `v2.0.129` | Build info |
| `instance` | pod name / hostname | Downward API |

`Bus.Members()` returns members with their tags; subscribers can filter (`role=ctrl`) for fan-out decisions. Queries take a tag filter so a "drain status" query from a ctrl pod only fans out to `role=api`.

Adding a new tag key is cheap, but every subscriber that filters on it must tolerate its absence on older nodes during rollout.

### Rollout

The current gossip cluster is not wired into any production code path, so there is no dual-publish phase and nothing to migrate at runtime.

1. Add `pkg/bus` implementing the `Bus` interface on top of Serf. Wire the cache invalidation subscriber.
2. Delete `pkg/cluster`, [`service-gossip-wan.yaml`](https://github.com/unkeyed/unkey/blob/main/k8s/api/base/service-gossip-wan.yaml), the WAN port (7947) from the deployment, and the bridge metrics in the same change.
3. Update the deployment to expose only the single gossip port and the per-region seed config; deploy region-by-region and watch membership convergence in Grafana.

### Observability

Carry over the existing metrics (`cluster_members_count`, `cluster_messages_received_total`, `cluster_message_latency_seconds`) but relabel by `topic` instead of `direction`. Drop `cluster_bridge_*` entirely. Add `bus_events_dropped_total{reason}` for the cases Serf surfaces explicitly (queue overflow, unknown topic, decode failure).

## Drawbacks

- **Larger blast radius for a misconfigured key.** One cluster means a bad rotation can partition every region simultaneously. Mitigate with staged rollouts of the secret and a kill-switch: a `Bus.Pause()` admin endpoint that stops publishing and unsubscribes all handlers without tearing down the process. (An env var would require a restart, which defeats the purpose during an incident.) The endpoint is bound to the pod's loopback interface only; authorization is by network policy, the same as the pprof port. It is not reachable from outside the node.
- **Serf is another dependency.** We already ship `memberlist`, so the marginal binary size is small, but it is one more upstream to track for CVEs.
- **Cross-region traffic costs.** A flat cluster gossips small heartbeats across regions continuously. With three regions and `WAN` timings this is on the order of KB/s per pod; trivial compared to API traffic but worth measuring.
- **Loss of the explicit bridge means no obvious "one place" to attach inspection tooling.** We trade a single chokepoint for uniform observability per-pod.

## Alternatives

- **Keep memberlist, add a real broker (NATS, Kafka).** Adds operational weight and a network round-trip we do not need. Gossip is fine for what we use it for.
- **Run Raft for everything.** Discussed above. Wrong consistency model for our workload, and forces a regional leader.
- **Keep the two-tier design but fix it.** We have iterated on the bridge twice already. The election, the WAN NLB, and the relay code are all coupled; surgical fixes leave us with the same surface area.
- **Embed an etcd client in every pod and use watches.** Operationally heavier than Serf, requires running an etcd cluster, and is overkill for fire-and-forget events.

## Unresolved questions

- Cross-region seed hostnames: do we use the existing headless services exposed via a peered ALB, or stand up a tiny dedicated `unkey-gossip-seed` service per region?
- Replay log size cap: 60 s of retention is fine for time, but we also need a per-pod byte cap so a hot publisher cannot OOM the ring buffer. Pick a default (suggest 16 MiB) and surface an overflow metric.
