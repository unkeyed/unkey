This guide covers how to apply database schema changes to the PlanetScale production database. You need this whenever a PR modifies the Drizzle schema files and is marked with **DB MIGRATION REQUIRED**.

## Prerequisites

- Access to the Unkey PlanetScale organization
- The `DRIZZLE_DATABASE_URL` connection string for the target database
- Node.js and pnpm installed locally
- The [unkey repository](https://github.com/unkeyed/unkey) cloned

## How the schema works

The database schema is defined using [Drizzle ORM](https://orm.drizzle.team/) in TypeScript. The source of truth is:

```
web/internal/db/src/schema/
```

Each file in that directory defines one or more tables. For example, `deployments.ts` defines the `deployments` table with its columns, indexes, and relations.

Two generated SQL files are derived from the Drizzle schema:

- `pkg/db/schema.sql` — used by the Go backend (sqlc)
- `pkg/mysql/schema.sql` — identical copy

These are regenerated by running `make generate-sql` and should never be edited manually.

## When to run a migration

Run a migration when a merged PR includes changes to files in `web/internal/db/src/schema/` and the PR description or comments indicate **DB MIGRATION REQUIRED**. Common changes include:

- Adding a new column to an existing table
- Adding a new enum value (e.g., adding `"stopped"` to a deployment status enum)
- Creating a new table
- Adding or modifying indexes

## Running the migration

<Steps>
  <Step title="Check out the correct branch">
    Make sure your local repository is on the branch or commit that contains the schema changes you want to migrate.

    ```bash
    git checkout main
    git pull origin main
    ```
  </Step>
  <Step title="Set the database URL">
    Export the PlanetScale connection string. You can find this in the PlanetScale dashboard under your database's connection settings.

    ```bash
    export DRIZZLE_DATABASE_URL="mysql://username:password@host/database?ssl={\"rejectUnauthorized\":true}"
    ```

    <Warning>
      Never commit this connection string to the repository. Use it only in your local terminal session.
    </Warning>
  </Step>
  <Step title="Navigate to the database package">
    ```bash
    cd web/internal/db
    ```
  </Step>
  <Step title="Install dependencies">
    ```bash
    pnpm install
    ```
  </Step>
  <Step title="Review the migration plan">
    Before applying changes, preview what Drizzle Kit will do:

    ```bash
    pnpm drizzle-kit push --dry-run
    ```

    Carefully review the output. It shows the SQL statements that will be executed against the database. Verify that:

    - Only expected tables and columns are being modified
    - No data-destructive operations (like dropping columns) are included unless intentional
    - Enum changes are additive (adding values, not removing)
  </Step>
  <Step title="Apply the migration">
    Once you've confirmed the changes look correct:

    ```bash
    pnpm run migrate
    ```

    This runs `drizzle-kit push`, which directly applies the schema diff to the connected database. Drizzle Kit compares the TypeScript schema definition against the live database and generates the necessary `ALTER TABLE` statements.

    <Info>
      PlanetScale handles schema changes with online DDL, so most migrations do not cause downtime. However, large table alterations may take time to complete.
    </Info>
  </Step>
  <Step title="Verify the migration">
    After the migration completes, verify the changes were applied. You can check through the PlanetScale dashboard or by running a query against the affected table to confirm the new columns or enum values exist.
  </Step>
</Steps>

## Common migration types

### Adding an enum value

When a new status or type value is added to a `mysqlEnum` column, the migration adds the value to the existing enum. For example, adding `"stopped"` to a deployment status:

```typescript
status: mysqlEnum("status", [
  "pending",
  "starting",
  "building",
  "deploying",
  "network",
  "finalizing",
  "ready",
  "failed",
  "skipped",
  "awaiting_approval",
  "stopped", // new value
])
```

This generates an `ALTER TABLE` that modifies the enum column to include the new value. Existing rows are unaffected.

### Adding a new column

New columns should have a default value or be nullable to avoid breaking existing rows:

```typescript
newColumn: varchar("new_column", { length: 256 }).default("some_default"),
```

### Creating a new table

New tables are straightforward. Drizzle Kit generates a `CREATE TABLE` statement. No existing data is affected.

## Troubleshooting

### Connection refused

If you get a connection error, verify:

- Your `DRIZZLE_DATABASE_URL` is correctly formatted
- Your IP is allowed in PlanetScale's access settings
- The SSL configuration is included in the connection string

### Unexpected changes in dry run

If `--dry-run` shows changes you didn't expect, your local schema may be out of sync. Pull the latest changes and try again:

```bash
git pull origin main
pnpm drizzle-kit push --dry-run
```

### Migration takes a long time

Large table alterations on PlanetScale may take time due to online DDL. This is normal for tables with millions of rows. Do not cancel the migration mid-process.
