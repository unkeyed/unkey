## DNS

Unkey uses latency-based geolocation routing to resolve both `*.unkey.app` [wildcard domains](/networking/wildcard-domains) and [custom domains](/networking/domains) to the closest region to the client.

## Gateway

The request arrives at the gateway, which terminates TLS close to the client so your app never needs to manage certificates. All connections enforce TLS 1.2 or higher, with TLS 1.3 preferred. HTTP requests are redirected to HTTPS automatically. Certificates are provisioned and renewed automatically for both wildcard and custom domains.

After terminating TLS, the gateway resolves the requested domain to a deployment by consulting a globally replicated metadata store. If the deployment runs in a different region, the gateway forwards the request to that region automatically.

The gateway then runs the policies configured for the deployment before the request reaches your code:

- [Authentication](/platform/gateways/authentication) verifies the caller's identity
- [Rate limiting](/platform/gateways/policies/rate-limiting) enforces request quotas
- [Custom policies](/platform/gateways/policies) apply additional rules

Requests that fail a policy receive an error response from the gateway without reaching your app.

After policies pass, the gateway selects a healthy instance of your deployment and proxies the request to your app. The response flows back through the gateway to the client.

### Request headers

The gateway adds headers to every proxied request so your app can identify the original client and request context:

| Header | Description |
| --- | --- |
| `X-Forwarded-For` | The original client IP address |
| `X-Forwarded-Host` | The original `Host` header from the client request |
| `X-Forwarded-Proto` | The protocol used by the client (`https`) |
| `X-Deployment-Id` | The deployment ID that the gateway resolved from the domain |
| `X-Unkey-Principal` | The authenticated principal, populated by the authentication policy |
