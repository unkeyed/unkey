A project is the top-level organizational unit for deploying applications on Unkey. Each project maps to a single codebase (typically a GitHub repository) and groups the [apps](/platform/apps/overview), [environments](/environments/overview), and configuration needed to build, deploy, and run your application.

## Project hierarchy

Projects live inside a workspace. A project contains one or more apps, and each app has its own environments, deployments, and configuration.

```text
Workspace
  └── Project (one per codebase)
        └── App (deployable service)
              ├── Environments (production, preview, custom)
              │     ├── Deployments
              │     ├── Variables
              │     └── Custom domains
              └── Configuration (build, runtime, gateway)
```

An [app](/platform/apps/overview) represents a single deployable service within your project. When you create a project, Unkey creates a default app automatically. A project can contain multiple apps, for example an API server and a background worker that share the same repository.

Build settings, runtime configuration, and [gateway policies](/platform/gateway/overview) are configured per app. [Variables](/platform/variables/overview) and [custom domains](/networking/domains) are scoped to an app's environment.

## Create a project

1. Navigate to your workspace's **Projects** page.
2. Click **New project**.
3. Enter a project name. Unkey generates a URL-friendly slug from the name, which you can edit.
4. Connect your GitHub account if you haven't already. You can skip this step and connect later.
5. Select the repository to deploy from your connected GitHub accounts. To finish onboarding without a repository, for example, if you plan to deploy with the [CLI](/build-and-deploy/cli) or want to connect GitHub later, click **Skip for now**.
6. Configure initial [build and runtime settings](/platform/projects/settings). You can change these later.
7. Click **Deploy** to trigger your first deployment.

The slug must be unique within your workspace. Dashboard-created slugs use lowercase letters, numbers, and hyphens.

If you skipped the repository step, the project is created without a connected repo. You can connect one later from the project's **Settings** tab and trigger your first deployment from there.

## Manage projects with the API

Use the Projects API when you need to provision, audit, or clean up projects from your own tooling. This is useful for internal developer platforms, customer onboarding flows, and scripts that keep Unkey projects in sync with another system.

The Projects API supports these operations:

| Operation | Endpoint | Required root key permission |
| --------- | -------- | ---------------------------- |
| Create a project | `POST /v2/projects.createProject` | `project.*.create_project` |
| List projects | `POST /v2/projects.listProjects` | `project.*.read_project` |
| Get a project | `POST /v2/projects.getProject` | `project.*.read_project` or `project.{projectId}.read_project` |
| Update a project | `POST /v2/projects.updateProject` | `project.*.update_project` or `project.{projectId}.update_project` |
| Delete a project | `POST /v2/projects.deleteProject` | `project.*.delete_project` or `project.{projectId}.delete_project` |

Project slugs are workspace-scoped handles that you choose. API-created slugs must be unique within the workspace and can contain letters, numbers, hyphens, and underscores. Get, update, and delete requests accept either the project ID or slug in the `project` field.

```bash
curl -X POST https://api.unkey.com/v2/projects.createProject \
  -H "Authorization: Bearer $UNKEY_ROOT_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Payments Service",
    "slug": "payments-service"
  }'
```

The response returns the generated project ID:

```json
{
  "meta": {
    "requestId": "req_1234abcd"
  },
  "data": {
    "id": "proj_1234abcd"
  }
}
```

Use delete protection when an automation can update project settings but must not delete the project without a separate step:

```bash
curl -X POST https://api.unkey.com/v2/projects.updateProject \
  -H "Authorization: Bearer $UNKEY_ROOT_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "project": "payments-service",
    "deleteProtection": true
  }'
```

Projects with delete protection enabled reject delete requests until you disable protection.

## Project dashboard

After creation, the project dashboard gives you four tabs:

| Tab            | What it shows                                                                                   |
| -------------- | ----------------------------------------------------------------------------------------------- |
| **Deployments** | Active and past deployments across all environments, with commit info and status                 |
| **Logs**        | Runtime logs from your running instances                                                        |
| **Requests**    | HTTP request analytics processed by [the gateway](/platform/gateway/overview)                   |
| **Settings**    | Build, runtime, and advanced configuration. See [Settings](/platform/projects/settings) for details |

The deployments tab is the default view when you open a project.

## Delete a project

<Warning>
  Deleting a project permanently removes all deployments, environments, custom domains, variables, and associated data. This action cannot be undone.
</Warning>

To delete a project:

1. Open the project actions menu (three-dot icon on the project card).
2. Select **Delete project**.
3. Check the confirmation box acknowledging the permanent deletion.
4. Confirm the deletion.

Deletion is asynchronous. Unkey removes all associated resources (deployments, environments, domains, variables) in the background. The project disappears from your workspace immediately, but cleanup of underlying resources completes shortly after.
