# Seeding the Unkey database

The schema is checked in at `pkg/mysql/schema/<table>.sql` (one file per table, kept in sync with the live database, regenerated by `make generate-sql`). The path is from the **repo root**: there is no `go/` prefix; `pkg/` lives at the top level. Read those files for columns, types, FKs, indexes, and check constraints. This skill covers what the schema can't tell you: the rules the application enforces in code that the database does not enforce in constraints. Violating them produces rows that pass FK checks but break the dashboard, the API, or the dataplane at runtime.

All table names below are the actual MySQL table names (snake_case).

## Foundations

- Every table has both `pk` (bigint autoincrement, the actual primary key) and `id` (varchar, unique, prefixed). FKs reference `id`, not `pk`. The MCP's `insert` tool auto-generates the prefixed `id` for known tables; pass it explicitly only when you need a fixed test ID.
- Junction tables (`keys_permissions`, `keys_roles`, `roles_permissions`, `encrypted_keys`) have no `id` column. Don't try to set one.
- For keys, always use the MCP's `generate_key` tool. It returns the `id` / `key` (plaintext) / `hash` / `start` set you need for the `keys` row. Don't compute hashes by hand.

## Always-create-together rules

These are the rules where one row alone is invalid in practice. Insert each group as a unit.

### Workspace

A workspace alone has no quota or rate-limit enforcement.

- `workspaces`
- `quota`: one row per workspace (1:1). Missing it breaks every per-workspace limit.

### API with key auth

The schema makes `apis.key_auth_id` nullable to support JWT auth, but for any seeding involving keys, always create the keyspace and link it.

- `key_auth`: the keyspace
- `apis` with `key_auth_id` pointing at the keyspace

### Application

An app without environments is invisible to the dashboard, and an environment without runtime/build settings can't deploy anything.

- `apps`
- Two `environments` rows: `slug='production'` and `slug='preview'`. Both reference the same `app_id`. The dashboard recognizes only these two slugs.
- One `app_runtime_settings` row per environment. Canonical defaults: `port=8080`, `cpu_millicores=250`, `memory_mib=256`, `storage_mib=0`, `command=[]`, `shutdown_signal='SIGTERM'`, `upstream_protocol='http1'`. `healthcheck` and `sentinel_config` can be empty.
- One `app_build_settings` row per environment. Canonical defaults: `dockerfile='Dockerfile'`, `docker_context='.'`, `watch_paths=NULL`.

### Deployable application (extends Application)

To make the app actually targetable for deployment to a region:

- All of the "Application" rows above
- One `app_regional_settings` row per (environment, region) pair you want to deploy to. The presence of this row IS the "deploy here" signal. Default `replicas=1`.

### Live deployment (extends Deployable application)

A `deployments` row with `status='ready'` (or any status where the deployment is meant to be actively serving) is broken without the rows below. The dataplane creates them in production; **in seeding, you must create them yourself, in the same unit**. Do not create a `ready` deployment and ask the user whether to add the companions; add them.

- `deployments`: the deployment itself, one per environment
- `deployment_topology`: one per region the deployment runs in
- `instances`: one per region, representing the actual running pods
- `cilium_network_policies`: one per region for in-cluster network access
- `frontline_routes`: at least one per deployment for edge reachability

For deployments in transitional or terminal statuses (`pending`, `building`, `deploying`, `failed`, `cancelled`, `stopped`, `superseded`, `skipped`, `awaiting_approval`), the companions are not required, since those statuses model states where the dataplane wouldn't have provisioned infrastructure yet, or has torn it down.

### Custom domain

The TLS verification pipeline expects both rows.

- `custom_domains`
- `acme_challenges`: paired 1:1 with the domain via `domain_id`

### Identity for rate limiting

A bare identity has no behavior distinct from a raw key. For a meaningful test:

- `identities`
- One or more `ratelimits` with `identity_id` set. `duration` is in milliseconds.

## Cross-row consistency

The schema lets you mix and match `workspace_id` values across an FK chain. The application doesn't.

- **`keys.workspace_id` MUST equal `key_auth.workspace_id`** for the referenced keyspace. Otherwise key verification routes to the wrong workspace and silently fails.
- **`ratelimits` rows** must set exactly one of `key_id` or `identity_id`. Never both. Never neither.
- **App / environment / project / workspace IDs** must be transitively consistent down the chain. Every row that carries `workspace_id` and `project_id` (environments, deployments, instances, sentinels, regional settings, etc.) must agree with its parent. When in doubt, copy the parent row's IDs into every child you create.
- **RBAC junction rows** (`keys_permissions`, `keys_roles`, `roles_permissions`) carry `workspace_id` directly. It must match the workspace of every entity they link.

## Conventions

- **Root workspace**: `ws_unkey` (hardcoded). The control-plane root API is `api_unkey`. Don't pick different IDs for these.
- **Environment slugs**: always `production` and `preview`. The dashboard recognizes nothing else.
- **Permission slug pattern**: `<resource>.<scope>.<verb>` (e.g. `api.*.create_key`, `rbac.*.read_role`). The `*` is a literal wildcard for the scope segment in seed/test data; production uses concrete IDs.
- **Region for local dev**: `name='local'`, `platform='dev'`. Upsert by `(name, platform)`; only one local region should exist.
- **Time fields**: most use millisecond Unix timestamps (column suffix `_at_m` or `_at`). `ratelimits.duration` is also milliseconds.

## Workflow

1. Read `pkg/mysql/schema/<table>.sql` (from the repo root, with no `go/` prefix) for any table you're about to insert into; schema wins over this skill if they conflict. For multi-table seeding, use `Glob`/`Read` to pull several files at once. Don't re-read a file you've already read this session.
2. Use the canonical defaults above for `app_runtime_settings` / `app_build_settings` / `app_regional_settings`. Don't grep the codebase for them; they're standard.
3. Compose the rows for one full unit (per the rules above) before calling `insert` or `bulk_insert`.
4. When inserting multiple rows into the same table (e.g., production+preview environments, or per-env settings), use one `bulk_insert` call instead of N `insert` calls. Saves round-trips.
5. Thread the returned `id` from each insert into the next as the FK value.
6. For `keys` rows: call `generate_key` first, then insert using the returned `id` / `hash` / `start` and the plaintext `key` for whatever consumer needs it.
7. Before reporting done, verify each unit is complete. If you created an `apps` row, both envs + 4 settings rows must exist. If you created a `ready` `deployments` row, its 4 companions must exist. **Don't ask the user "should I add the missing rows?"; finish the unit you started.** Asking instead of completing is the most common failure mode.

## Maintenance

If you change the Unkey schema, update this skill in the same PR. The DDL files at `pkg/mysql/schema/` are authoritative, but the rules here drift silently. When they drift, the agent confidently produces broken rows.
